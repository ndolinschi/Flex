# Конкурентный анализ агентов — июль 2026

Отчёт подготовлен для проекта **Flex** — open-source agent-loop engine на Rust (pre-alpha v0.1.0, MIT/Apache-2.0). Все нетривиальные утверждения снабжены ссылками на источники (раздел «Источники»); утверждения, опровергнутые при фактчекинге, скорректированы; непроверяемые помечены как «по неподтверждённым данным».

## Executive summary

Рынок агентных систем в середине 2026 года разделился на четыре лагеря. **Продукты-агенты для разработчиков** (OpenCode, ~184k звёзд; Devin от Cognition, оценка $25B pre-money) конкурируют за рабочий процесс инженера: OpenCode выигрывает открытостью и client/server-архитектурой с OpenAPI-сервером, Devin — глубиной оркестрации («Devin Manages Devins», изолированные VM на сессию, стек организационной памяти Knowledge/Playbooks/Skills). **Персональные агенты-демоны** (OpenClaw, ~380k звёзд — самый быстрорастущий репозиторий GitHub; Hermes Agent от Nous Research, ~185–211k звёзд) показали взрывной спрос на always-on ассистентов в мессенджерах — и одновременно катастрофические провалы безопасности: 512 уязвимостей в аудите OpenClaw, 36,8% skills в ClawHub с проблемами безопасности (исследование Snyk ToxicSkills), неаутентифицированный RCE в OpenCode (CVE-2026-22812). **Фреймворки-инфраструктура** (LangChain ~141k звёзд, LangGraph ~36,8k) закрепились в enterprise через durable execution, checkpointing и open-core-модель LangSmith. Наконец, **паттерны self-improving систем** вокруг Claude Fable 5 (запуск 9 июня 2026) оформились в воспроизводимую дисциплину: независимые верификаторы со свежим контекстом ловят ~73% подсаженных дефектов против 7–33% у самокритики, память компаундируется в файлах (STATE.md, Skills), а не в весах модели.

Ключевые выводы для Flex. (1) Ниша Flex — «полиглотная» оркестрация: ни один из шести конкурентов не позволяет использовать чужие агенты (Claude Code, Cursor, opencode, Copilot, ACP) как взаимозаменяемые сабагенты через единый трейт — это реальная, незанятая дифференциация. (2) Худшее слабое место всего рынка — безопасность и изоляция: worktree-изоляция + executor-бэкенды с network policy + injection-scan hooks у Flex уже сейчас сильнее, чем у OpenClaw, Hermes и OpenCode. (3) Главные пробелы Flex на момент написания отчёта: нет durable checkpointing уровня LangGraph, нет client/server-разделения уровня OpenCode, нет продуктовой зрелости, экосистемы и дистрибуции. (4) Приоритетные заимствования: verifier-сабагенты «maker ≠ grader», checkpointer-абстракция для session actor, headless-сервер с OpenAPI, каналы мессенджеров для packages/gateway — с уроками безопасности OpenClaw, выученными заранее.
>
> **Обновление 2026-07-09**: пункты (3)–(4) закрыты, кроме каналов мессенджеров и eval-harness — см. «Статус» под каждой рекомендацией в разделе «Что позаимствовать» ниже. Checkpointing, client/server-сервер, независимый верификатор, goal-loops, HITL-гейт на память, декларативные workflow-пайплайны и cron/webhook-раутины реализованы; экосистема/дистрибуция остаются открытым пробелом, как и были.

## Профили конкурентов

### 1. OpenCode (SST)

**Позиционирование.** Open-source (MIT) AI coding agent, провайдер-нейтральная альтернатива Claude Code. Terminal-first (TUI) + десктоп-приложения (macOS/Windows/Linux), IDE/VSCode-расширение и headless server/SDK — одновременно CLI-продукт, рантайм и интеграционная платформа. Инструмент бесплатен; монетизация через **OpenCode Zen** — опциональный pay-as-you-go шлюз курируемых моделей (~$0,20–$30 за 1M входных токенов) с командными workspace, RBAC и лимитами расходов *(подтверждено фактчекингом)*. Сигналы принятия очень сильные: ~184k звёзд GitHub, 22,8k форков, 832 релиза, v1.17.15 от 7 июля 2026 *(подтверждено)*.

**Архитектура.** Клиент-серверный дизайн: `opencode serve` поднимает headless HTTP-сервер с OpenAPI 3.1 (по умолчанию 127.0.0.1:4096, опциональная парольная аутентификация) — TUI, IDE-плагины и программные клиенты работают с одним бэкендом; есть сгенерированный типобезопасный SDK *(подтверждено)*. Agent loop: primary-агенты Build (полный доступ к инструментам) и Plan (правки/bash требуют одобрения), сабагенты General (полный доступ), Explore (read-only анализ кода), Scout (read-only внешний ресёрч); скрытые системные агенты для компакции и суммаризации. Агенты настраиваются через JSON/Markdown: модель, temperature, max steps, allow/ask/deny-права с glob-паттернами на каждого агента. Инструменты: встроенные + кастомные через плагины (Zod-схемы), MCP, LSP-интеграция для диагностик. Изоляция — permission-based, **без жёсткого sandbox**; CVE-2026-22812 (CVSS 8.8, неаутентифицированный RCE до v1.1.10) показал риск открытого сервера. Кодовая база: TypeScript/Bun-монорепо.

**Сильные стороны.**
- Единственная среди терминальных агентов настоящая client/server-архитектура с OpenAPI-сервером и SDK — уникально скриптуема и встраиваема.
- Провайдер-нейтральность: любой LLM-провайдер + шлюз Zen; отсутствие vendor lock-in (MIT).
- Лучший в классе TUI (темы, undo/redo, drag-and-drop изображений) + десктоп + IDE-расширение.
- Богатая расширяемость: event-hook плагины (локальные или npm), кастомные агенты/команды/skills, MCP.
- Тонкая пер-агентная permission-модель и read-only сабагенты.
- Drop-in совместимость с экосистемой Claude: читает `.claude/skills` и конвенцию AGENTS.md *(подтверждено)*.

**Слабые стороны.**
- Медленнее и «прожорливее» Claude Code в лобовых тестах (27 мин / 278k токенов против 14 мин / 191k на той же задаче Opus 4.5; в другом тесте ~78% дольше).
- Жалобы на качество кода: пропущенные импорты, чрезмерно агрессивные переименования, самовольный реформат — шумные диффы и техдолг.
- Нестабильность из-за очень быстрого релизного цикла (3,6k открытых issue).
- Инцидент CVE-2026-22812; нет жёсткого sandbox — безопасность держится на permission-промптах.
- Меньше выгоды от провайдер-специфичных оптимизаций (кеширование, тюнинг), чем у first-party инструментов.

**Self-improvement.** Накопление опыта конвенционное и файловое, не автономное: AGENTS.md (генерируется `/init`, коммитится в git), нативная система skills (SKILL.md с YAML-frontmatter в `.opencode/skills/`, `~/.config/opencode/skills/` и Claude-совместимых путях, подгружаются on demand нативным инструментом `skill`, гейтятся пер-агентными правами) — skills пишут люди, не агент. Компакция сессий с настраиваемыми суммари. Самокоррекция только на уровне loop: LSP-диагностики в цикле, Plan-then-Build, undo/redo, лимиты max-steps. Автоматической записи памяти, выученных рутин или подсистемы-верификатора в документации не обнаружено.

### 2. Hermes / Nous Research

**Позиционирование.** Полный открытый стек, а не один продукт: (1) **Hermes 3/4** — open-weight, «нейтрально выровненные» модели; (2) **Hermes Agent** (февраль 2026, MIT) — self-hosted автономный агент-рантайм («агент, который растёт вместе с вами»), работает от VPS за $5 до GPU-кластеров, доступен через 20+ мессенджеров; (3) **Atropos** (MIT) — фреймворк RL-окружений для тренировки tool use/reasoning; (4) Forge Reasoning API; (5) Nous Portal — платный хостинг инференса. Hermes Agent имел ~185k звёзд к июню 2026 (211k при более позднем просмотре) *(подтверждено)*. **Важная поправка фактчекинга:** Hermes 4 — гибридное reasoning-семейство (переключаемый `<think>`), но только 70B и 405B являются post-train'ами Llama 3.1 под лицензией Llama 3; вариант 14B основан на **Qwen3-14B и выпущен под Apache 2.0** (добавлен после первоначального запуска 70B/405B).

**Архитектура.** Ядро — сам agent loop как синхронный движок оркестрации (в отличие от gateway-first дизайна OpenClaw). Память: агент-курируемая персистентная память с FTS5 полнотекстовым кросс-сессионным поиском, LLM-суммаризация, session notes, опциональное Honcho-моделирование пользователя. Изоляция: **шесть подключаемых терминальных бэкендов — local, Docker, SSH, Daytona, Singularity, Modal**; модель безопасности считает ОС единственной границей сдерживания. Мультиагентность: изолированные сабагенты для параллельных потоков работы; cron-задачи в свежих сессиях. Инструменты: 60+ встроенных, MCP, программный вызов инструментов через execute_code. Провайдеры: model-agnostic — Nous Portal, OpenRouter, OpenAI, любой OpenAI-совместимый endpoint. Тренировочная сторона: Atropos собирал RL-траектории (tool-calling, code-exec, мультимодальные окружения) с интеграциями Axolotl/vLLM/SGLang.

**Сильные стороны.**
- Вертикально интегрированный открытый стек: одна организация тренирует open-weight модели (Hermes 4 + Atropos RL) и выпускает рантайм, который их эксплуатирует.
- Взрывное принятие (~185–211k звёзд), позиционируется как первая реальная альтернатива OpenClaw.
- Отличительный self-improvement-цикл: автоматическое превращение успешных workflow в переиспользуемые skills *(подтверждено)*.
- Дешёвый гибкий деплой: от VPS за $5 до serverless/GPU, 20+ чат-платформ из одной инсталляции.
- Полностью открытые лицензии: MIT-рантайм, открытые веса, 94-страничный техотчёт Hermes 4 (arXiv 2508.18255).

**Слабые стороны.**
- Слабое сдерживание: ОС — единственная граница безопасности против враждебного LLM; API-сервер допускает неаутентифицированный доступ при незаданном API_SERVER_KEY на 0.0.0.0.
- Напряжение в governance: ~18 700 открытых issue хоронят security-репорты; Nous отредактировала GitHub-issue, удалив претензии о плагиате, — критика на Hacker News.
- **Atropos заархивирован (read-only) 4 июля 2026** — RL-половина стека без сопровождения *(подтверждено)*.
- Веса 70B/405B под лицензией Llama 3 (не полностью пермиссивной), это post-train'ы, а не модели «с нуля».
- «Неограниченные» модели + автономно самомодифицирующиеся skills — проблема для enterprise-комплаенса.

**Self-improvement — ключевой дифференциатор.** Замкнутый цикл обучения: (1) автономное создание skills — агент оценивает собственные взаимодействия и кодирует эффективные workflow как переиспользуемые skills; (2) ревизия существующих skills прямо во время использования; (3) стандарт agentskills.io, обмен через Skills Hub *(подтверждено)*; (4) персистентная память (FTS5-поиск, суммаризация, «knowledge persistence nudges» — периодические подталкивания к консолидации); (5) Honcho-моделирование пользователя. Медленный контур — Atropos: RL-окружения генерировали траектории для тренировки tool use в весах моделей (улучшение и in-weights, и in-context/on-disk). Монетизация: Nous Portal — Free / $20 / $100 / $200 в месяц с соответствующими кредитами; Hermes 4 405B — около $1,00/1M входных токенов (по неподтверждённым данным в части точных цен API).

### 3. Devin (Cognition)

**Позиционирование.** Проприетарный, закрытый SaaS «AI software engineer» — полностью автономный облачный агент (не фреймворк), доступ через app.devin.ai, Slack/Teams/Linear/Jira, agent-native облачную IDE, CLI и v3 REST API. Биллинг: подписка + usage-based ACU (Agentic Compute Unit ≈ 15 минут активной автономной работы). Финансовая динамика: покупка Windsurf (июль 2025, ~$250M по сообщениям, $82M ARR, 350+ enterprise-клиентов), раунд $400M при оценке $10,2B (сентябрь 2025), $1B при $25B pre-money (май 2026); ARR более чем удвоился после поглощения *(подтверждено)*.

**Ценообразование — поправка фактчекинга.** Снижение входной цены с $500/мес до $20 с Devin 2.0 (апрель 2025) подтверждено, и структура Core ($20 + $2,25/ACU) / Team ($500/мес, 250 ACU по $2,00) существовала, но **на июль 2026 актуальные планы иные**: Free ($0), Pro ($20/мес), Max ($200/мес), Teams ($80/мес + $40/место), Enterprise (custom, биллинг в ACU). Конкурентные сессии: до 10 на Free/Pro, безлимит на Max/Teams/Enterprise; квоты обновляются ежедневно/еженедельно, overage по API-ценам.

**Архитектура.** Каждая сессия — собственная изолированная облачная VM с полной dev-средой: терминал, браузер, редактор, запуск тестов, «computer use» для десктопного end-to-end QA (февраль 2026). Interactive Planning (Devin 2.0): анализ кодовой базы и редактируемый план перед автономным исполнением. **«Devin Manages Devins» (GA 19 марта 2026)**: координатор-Devin скоупит работу, порождает дочерних Devin'ов (каждый — полный Devin в своей изолированной VM), шлёт корректировки mid-task, мониторит расход ACU, ставит на паузу/убивает отстающих и **читает полные траектории детей**, чтобы понять, что сработало *(подтверждено; поправка: structured output schemas — отдельная, ранее существовавшая возможность API, а не заявленный элемент этой фичи)*. Devin Security Swarm (1 июля 2026): agentic map-reduce, каждая находка воспроизводится в изолированной песочнице для подтверждения эксплуатируемости. Cognition тренирует собственную линейку SWE-моделей (SWE-1.5 → SWE-1.6, ~950 ток/с). Экосистема MCP: 90+ коннекторов, персональные MCP-серверы, enterprise-allowlist'ы. «Devin 3» как продукт не существует — официальные релизы 2026: Devin 2.2 и V3 API.

**Сильные стороны.**
- Глубочайшая продуктовая мультиагентная оркестрация: координатор + дочерние VM, mid-task steering, мониторинг ACU, ревью траекторий.
- Полная изолированная VM на сессию с реальной верификацией (тесты, браузер, computer-use QA), а не diff-only генерация.
- Непревзойдённый стек организационной памяти: Knowledge (авто-вспоминаемый контекст), Playbooks, Skills, авто-генерируемый Repo Knowledge, DeepWiki с архитектурными диаграммами.
- Сильный enterprise-governance: SSO/IdP, enterprise-секреты, лимиты ACU с аудитом, MCP-allowlist'ы.
- End-to-end асинхронный workflow (Slack-синхронизация тредов, Linear/Jira/GitLab/GitHub Enterprise, автоматизации, v3 API) — работает как удалённый коллега.
- Огромная капитализация и импульс ($25B pre-money, май 2026).

**Слабые стороны.**
- Исторически слабая реальная автономность: оценка Answer.AI (январь 2025, Devin 1.0) — 14 провалов против 3 успехов на 20 реальных задачах; исходные 13,86% SWE-bench считались на 25%-м сабсете; обновлённых цифр для 2.x нет.
- Непредсказуемость затрат: ACU-метеринг превращает вход за $20 в $300–500/мес на реальной нагрузке; Fast Mode за 4x ACU усугубляет.
- Смешанная удовлетворённость (~3,0/5 на Trustpilot, март 2026): необъяснимые провалы задач, медленный вывод.
- Cloud-only, закрытый, проприетарный: без self-hosting, lock-in на VM и ACU-биллинг Cognition.
- Разработчики чувствуют себя отстранёнными от архитектурных решений; лучше всего на хорошо скоупленных задачах.

**Self-improvement.** Многослойный и явный (курируется человеком, не автономен): Knowledge — персистентное хранилище подсказок/конвенций, авто-вспоминаемое во всех сессиях; Devin проактивно предлагает новые записи, пользователь одобряет. Repo Knowledge — авто-скан репозиториев. Playbooks — переиспользуемые пошаговые промпты. **Skills (13 февраля 2026)** — переиспользуемые SKILL.md-инструкции, которые Devin активирует и вызывает *(подтверждено; оговорки фактчекинга: в первичных доках вызов через авто-активацию или `@skills:name`, slash-команды документированы только для Devin CLI по сторонним источникам; skill-специфичная телеметрия — по неподтверждённым данным)*. DeepWiki/Devin Wiki — автоматическая индексация репо в документацию раз в две недели; Wiki v2 (апрель 2026) с сабагентами. Внутрисессионная самокоррекция: планы декомпозируются в верифицируемые шаги, дочерние агенты сами гоняют тесты перед отчётом.

### 4. OpenClaw (экс-Warelay → Clawdbot → Moltbot)

**Позиционирование.** Бесплатный MIT-лицензированный персональный AI-ассистент/рантайм, self-hosted, local-first. Не coding CLI, а always-on агент-демон с UI в привычных мессенджерах. Целевая аудитория — технически подкованные пользователи; мейнтейнер публично предупреждал, что для людей без CLI-навыков это «слишком опасно». Бизнес-модели нет: пользователи приносят свои LLM-ключи. История имён *(подтверждено)*: Warelay (ноябрь 2025) → Clawdbot (2 января 2026) → Moltbot (27 января 2026, после трейдмарк-претензий Anthropic) → OpenClaw (30 января 2026). Создатель Peter Steinberger ушёл в OpenAI 14 февраля 2026, стюардство — у некоммерческого OpenClaw Foundation.

**Архитектура.** Ядро — local-first Gateway-демон, control plane для сессий, каналов, инструментов и событий *(подтверждено)*. TypeScript/Node.js, pnpm-монорепо, Docker, companion-приложения (macOS/Windows/iOS/Android). Каналы: 20+ мессенджеров (WhatsApp, Telegram, Slack, Discord, Signal, iMessage, Teams, Matrix, IRC, WeChat, QQ, LINE, Nostr, Twitch и др.) + голос + Canvas. Мультиагентность: входящие каналы/аккаунты/пиры маршрутизируются на изолированные агенты с отдельными workspace. Skills — Markdown-пакеты SKILL.md по спецификации AgentSkills с иерархией загрузки (workspace > project > personal > managed > bundled). Конфиг workspace в `~/.openclaw/workspace` с инжектируемыми промпт-файлами (AGENTS.md, SOUL.md, TOOLS.md). Model-agnostic (OpenAI GPT/Codex, Claude, DeepSeek). Sandboxing — известное слабое место: аудиты критиковали широкие системные права.

**Сильные стороны.**
- Самое широкое покрытие мессенджеров среди персональных агентов (20+ платформ).
- Local-first, self-hosted, MIT — полное владение данными, без lock-in.
- Взрывное комьюнити: самый быстрорастущий репозиторий GitHub, ~380k звёзд, 80k форков, 900+ контрибьюторов, маркетплейс skills (ClawHub).
- Прозрачная plain-Markdown память («no hidden state») с гибридным семантическим/keyword-поиском и подключаемыми бэкендами (SQLite, LanceDB, Honcho, QMD) *(подтверждено)*.
- Простая расширяемость: skills — просто Markdown-файлы, агенты сами могут черновики новых способностей.
- Сильная экосистемная тяга: сервисы на базе OpenClaw от Tencent и Z.ai, адаптация к DeepSeek/WeChat в Китае.

**Слабые стороны.**
- Тяжелейший security-послужной список: аудит конца января 2026 — 512 уязвимостей (8 критических); ~1 000 неаутентифицированных инстансов в Shodan; продемонстрирована prompt-injection-эксфильтрация SSH-ключей/токенов через одно письмо или веб-страницу (демо Giskard, Archestra).
- Токсичная supply chain skills: исследование Snyk ToxicSkills по 3 984 skills в ClawHub/skills.sh — 36,8% с проблемами безопасности, 13,4% с критическими, 76 подтверждённых вредоносных payload'ов; публикация в ClawHub требует лишь SKILL.md и недельного GitHub-аккаунта.
- Широкие системные права по умолчанию; слабый sandboxing.
- Governance-инциденты: MoltMatch (февраль 2026) — агенты автономно создавали профили на сайтах знакомств без авторизации пользователя.
- Регуляторный удар: Китай запретил использование госорганами и банками (март 2026).
- Неопределённость после ухода создателя: проект зависит от молодого OpenClaw Foundation.

**Self-improvement.** Файловый и явный *(подтверждено)*: MEMORY.md (долговременные факты, загружаются на старте сессии), дневные заметки memory/YYYY-MM-DD.md (индексируются для поиска), опциональный DREAMS.md. Перед компакцией контекста — «тихий ход», побуждающий агента сохранить важное в файлы памяти. Инструменты memory_search (гибридный поиск) и memory_get. Опциональный «Dreaming» — фоновая консолидация, продвигающая записи в MEMORY.md по частоте/релевантности — настоящий кросс-сессионный контур дистилляции. Skill Workshop — очередь предложений: агенты пишут черновики новых skills, человек-оператор ревьюит перед включением. Персона эволюционирует через редактируемые SOUL.md/AGENTS.md. Встроенного автоматического верификатора нет — коррекция через ревью оператора и записи в память.

### 5. LangChain (эпоха 1.0) + LangSmith, Deep Agents

**Позиционирование.** Open-source агентный фреймворк (Python и JS), MIT, от LangChain Inc. («the agent engineering platform»). После 1.0 (GA 22 октября 2025): LangChain — «самый быстрый способ собрать агента», высокоуровневый provider-agnostic harness вокруг `create_agent`; LangGraph — низкоуровневый оркестрационный рантайм (create_agent работает поверх него) *(подтверждено)*. Бизнес-модель open-core: бесплатный OSS + LangSmith (observability/evals/deployment SaaS) + no-code Agent Builder. Series B $125M при оценке $1,25B (IVP, октябрь 2025) *(подтверждено)*; ~$16M ARR по сторонним оценкам (по неподтверждённым данным). ~141k звёзд, v1.3.11 (июнь 2026) *(подтверждено)*.

**Архитектура.** `create_agent` реализует стандартный ReAct-цикл («модель вызывает инструменты в цикле до завершения задачи») поверх graph API LangGraph — наследует durable execution: checkpointer'ы персистят состояние, thread_id-scoped кратковременная память, first-class human-in-the-loop interrupts. Кастомизация через композируемые **middleware** (HITL-одобрение, суммаризация у лимита токенов, PII-редактирование; категории: execution environment, context management, planning, fault tolerance, guardrails, human steering) *(подтверждено — три столпа 1.0: create_agent, middleware, стандартные content blocks)*. Провайдеры — ключевой дифференциатор: модели задаются строками «provider:model», стандартизованные content_blocks как provider-agnostic спецификация (reasoning traces, citations, server-side tool use). Мультиагентность: графы LangGraph + пакет **deepagents** («batteries-included harness»: planning/todo-инструменты, виртуальная ФС, порождение сабагентов, долговременная память). Sandboxing в OSS-ядре отсутствует; LangSmith продаёт hosted Sandboxes. Legacy AgentExecutor и langgraph.prebuilt депрекейтнуты.

**Сильные стороны.**
- Крупнейшая интеграционная экосистема (сотни моделей/vectorstore/инструментов), настоящая подмена провайдеров строковыми идентификаторами.
- Production-grade ReAct-цикл на durable-рантайме LangGraph с минимумом кода.
- Middleware — чистые точки расширения вместо форка фреймворка.
- Полный коммерческий стек: LangSmith tracing/evals/deploy, one-click deploy.
- Слоистое покрытие всего спектра: no-code Agent Builder → deepagents → create_agent → низкоуровневые графы.
- Обязательство стабильности 1.0 (без breaking changes до 2.0).

**Слабые стороны.**
- Многолетняя критика «abstraction bloat»: вложенные абстракции, тяжёлый дебаг; знаменитый разбор Octomind «Why we no longer use LangChain».
- История ломающих изменений и фрагментированной документации; дефицит доверия сохраняется и после 1.0.
- Overkill для простых RAG/LLM-приложений (базовый RAG-пайплайн — 80+ строк против ~30 в LlamaIndex).
- Продуктовый разброс: LangChain vs LangGraph vs deepagents vs LCEL vs депрекейтнутый AgentExecutor — непонятно, какой слой брать.
- Нет встроенного sandbox/изоляции в OSS; безопасное исполнение — платная фича LangSmith.
- Баги от быстрой итерации (например, create_agent response_format хардкодил tool_choice, ломая Anthropic thinking + structured output в v1.1–1.12).

**Self-improvement.** Через инфраструктуру памяти, а не автономную систему skills. LangGraph Store — долговременная память как JSON-документы по namespace/key. **LangMem SDK** (начало 2025): извлечение фактов/поведения из разговоров и оптимизация поведения агента через автоматические обновления промптов — алгоритмы metaprompt, gradient, prompt_memory. deepagents встраивает долговременную память и планирование. Дополняющий контур — LangSmith: production-трейсы питают датасеты evals («continuous testing and improvement») — human/eval-driven цикл, а не автономное приобретение skills. Маркетплейса skills или самописных рутин в OSS-фреймворке нет. Ценообразование LangSmith *(подтверждено)*: Developer free (5k трейсов/мес), Plus $39/место/мес, трейсы $2,50/1k (14 дней) и $5,00/1k (400 дней).

### 6. LangGraph (LangChain Inc.)

**Позиционирование.** Open-source, MIT, «низкоуровневый оркестрационный фреймворк для stateful-агентов». Библиотека/рантайм (Python + LangGraph.js), open-core: бесплатный OSS + платный LangGraph Platform (managed-деплой, GA с мая 2025) + LangSmith. Целевая аудитория — команды, строящие production-агентов с длительным временем жизни. Заявленные пользователи: Klarna, Replit, Elastic; анонс 1.0 называет Uber, JP Morgan, BlackRock, Cisco и 90M ежемесячных загрузок по экосистеме (цифра — заявление вендора). ~36,8k звёзд, v1.2.8 от 6 июля 2026 *(подтверждено)*.

**Архитектура.** Ядро — явный state graph: типизированная схема состояния, узлы-функции, (условные) рёбра; циклы дают агентные loops. **Durable execution через подключаемые checkpointer'ы** (InMemorySaver, SqliteSaver, PostgresSaver; комьюнити Redis/DynamoDB), персистящие состояние треда как чекпоинты; возобновление — повторный вызов с тем же thread_id *(подтверждено)*. Human-in-the-loop через interrupts: пауза, инспекция/правка состояния, продолжение с чекпоинта. Память двухуровневая: thread-scoped кратковременная (checkpointer) + долговременный Store API — JSON-документы в иерархических namespace с семантическим поиском, доступные из любого треда. Мультиагентные паттерны: supervisor/иерархия, swarm/handoffs, подграфы-как-агенты. Sandboxing/изоляции кода нет — безопасность инструментов на разработчике; OSS-рантайм однопроцессный (worker pool/task queue — в платном Platform).

**Сильные стороны.**
- Самая зрелая история durable execution/checkpointing среди OSS-фреймворков: подключаемая персистентность, resume по thread_id, time travel, first-class HITL.
- Тонкий явный контроль потока (типизированное состояние, условные рёбра, подграфы) — подходит регулируемым отраслям; enterprise-принятие тому свидетельство.
- Встроенная двойная модель памяти + процедурная память: агент хранит собственный системный промпт в Store, и шаг рефлексии переписывает инструкции — агенты «программируют сами себя» без изменения весов.
- MIT + щедрый бесплатный self-hosted-тариф; огромная экосистема провайдеров через интеграции LangChain.
- Гарантия стабильности 1.0 (октябрь 2025) *(подтверждено)*; плавная эскалация от create_agent до сырых графов.

**Слабые стороны.**
- Крутая кривая обучения и boilerplate (схемы состояния, узлы, рёбра) — over-engineering для простых агентов.
- Checkpointing — это «save points», а не полноценный durable execution: нет автодетекции сбоев и автоматического возобновления, нет координации дублирующихся resume, однопроцессный OSS-рантайм (критика Diagrid) — полная надёжность требует платного Platform или внешней инфраструктуры типа Temporal.
- Тяжёлый дебаг глубоких/вложенных графов и «тихих» логических сбоев; latency сериализации состояния на масштабе.
- Историческая репутация API-churn и устаревших туториалов (смягчено после 1.0).
- Нет встроенного sandboxing, верификатора или механизма библиотеки skills — self-improvement собирается вручную (или через LangMem).

**Поправка фактчекинга по ценам Platform.** Developer-тариф описан верно (бесплатный self-hosted до 100k node executions/мес), но Plus **не включает** 100k executions: Plus требует подписку LangSmith Plus ($39/пользователь/мес) и тарифицирует $0,001 за node execution с первого узла (плюс standby ~$0,0007–$0,0036/мин); к середине 2026 деплой LangGraph вообще свёрнут в единый прайсинг LangSmith ($0,005/run + LCU-метеринг движка).

## Паттерны self-improving систем (по материалам статьи 0xCodez и серии loop-engineering-гайдов, июнь–июль 2026)

Контекст: **claude-fable-5** запущен 9 июня 2026 как первая публичная модель Mythos-класса, рассчитанная на многодневную работу. Цена $10/M input, $50/M output — **2x от Opus 4.8** ($5/$25), а не ~5x, как утверждал исходный материал *(поправка фактчекинга)*; 90% скидка на prompt-cache reads.

**Центральный тезис: self-improving ≠ self-learning.** Ни одна публичная production-модель не обновляет веса из ваших сессий. «Самоулучшение» — компаундирование системы: STATE-файлы, Skills, eval-циклы, при неизменной модели-оркестраторе.

**Шестичастный цикл (ядро 14-шагового фреймворка):**
триггер (cron / падение CI / webhook / self-paced) → загрузка правил (замороженный CLAUDE.md + накопленная память) → executor (один ограниченный кусок работы) → **верификатор** (сабагент со свежим контекстом, оценивает только артефакты + рубрику, никогда не видит рассуждения исполнителя) → запись в память → stop check (done / escalate / park / go again).

**Три оркестрационных примитива:**
1. **Goal-loops:** `/goal` (Claude Code v2.1.139+, остановка по условию, минуты–часы) против Outcomes для Managed Agents (часы–дни); `/loop` (v2.1.72+) для регулярных прогонов, `/schedule` для облачных Routines.
2. **Независимые верификаторы — «maker is never the grader».** Верификаторы со свежим контекстом ловят ~73% подсаженных дефектов против 7–33% у самокритики и дали ~6x больше улучшения пайплайна, чем самокритика Opus 4.7, в эксперименте Parameter Golf (8xH100, до 8 часов) *(подтверждено, но это один-единственный эксперимент — экстраполировать осторожно)*. Вариант — vision-верификация: maker рендерит скриншот UI, верификатор сравнивает с целью, design tokens и предыдущим скриншотом из STATE.md.
3. **Dynamic Workflows** (Claude Code, 28 мая 2026 *(подтверждено)*): модель сама пишет кастомный JS-harness с `agent()`, `parallel()`, `pipeline()` — fan-out-and-synthesize, adversarial verification, loop-until-done, вместо фиксированных графов workflow.

**Память — пятиступенчатая прогрессия:** задокументировать провал → диагностировать корневую причину → превратить догадку в проверенный факт → дистиллировать в общее правило → консультироваться с правилом на старте сессии. Носители: STATE.md (разделы Verified facts / General rules / Open failures / Lessons learned / Last session), memory/INDEX.md (запись каждый цикл — урок цикла 3 доступен циклу 40), Skills в `~/.claude/skills/` как процедурная память, переживающая проекты.

**Routines** (research preview 14 апреля 2026 *(подтверждено)*): сохранённые конфигурации исполняются в облаке Anthropic по расписанию, API- или GitHub-триггерам — компаундирование становится непрерывным, laptop-off. Осторожно: на июль 2026 это не-GA инфраструктура.

**Экономика и safety:** маршрутизация по ярусам моделей (Fable 5 — план/ревью на xhigh, Sonnet — имплементация, Haiku 4.5 — грейдинг, Opus 4.8 — сложные подзадачи и фолбэк при срабатывании классификаторов) примерно вдвое режет стоимость фичи. Production-безопасность циклов — явные stop-правила: доказательство успеха, потолок «3 одинаковых провала», бюджеты токенов. Изоляция параллельных агентов — git worktrees (та же модель, что у Flex).

**Ограничения источников:** материалы — энтузиастский/промо-контент (Medium/Substack) с минимумом критической оценки; память — plain-text конвенция без механизма, мешающего компаундированию устаревших «проверенных фактов».

## Чем Flex отличается

### Реальные уникальные преимущества

**1. Полиглотная оркестрация внешних агентов — незанятая ниша.** Ни один из шести конкурентов не решает задачу «чужие агенты как взаимозаменяемые сабагенты». Devin оркестрирует только Devin'ов; OpenCode, Hermes и OpenClaw гоняют собственные сабагенты; LangChain/LangGraph оркеструют LLM-вызовы и инструменты, но не полноценные внешние агент-CLI. Единый Agent-трейт Flex, где нативный loop и Claude Code/Cursor/opencode/Copilot/ACP взаимозаменяемы и вкладываются друг в друга, — это дифференциация, которую подтверждает рыночный тренд: `grid:coding-agents-farm`-подобные сценарии («ферма» разных агентов на изолированных worktree) уже востребованы, но собираются скотчем поверх CLI. Flex делает это контрактом первого класса.

**2. Нормализованный event stream + JSONL ground truth.** Ближайший аналог — стандартные content blocks LangChain (нормализация вывода провайдеров), но Flex нормализует весь жизненный цикл агента (включая ToolCall с permission trail), а append-only JSONL даёт воспроизводимый ground truth, которого нет ни у кого: у Devin траектории проприетарны, у OpenCode события завязаны на его сервер.

**3. Executor-бэкенды с network policy.** Прямой аналог — 6 бэкендов Hermes Agent (local/Docker/SSH/Daytona/Singularity/Modal), но у Hermes «ОС — единственная граница» и нет network policy. Flex (local sh, docker, ssh+rsync, apptainer, serverless stub + network policy) — единственный, кто совмещает подключаемость исполнения с политикой сети. На фоне CVE OpenCode, 512 уязвимостей OpenClaw и ToxicSkills это сильный аргумент.

**4. Worktree-изоляция + per-turn snapshots.** У Devin есть VM-изоляция (сильнее, но cloud-only и проприетарно), у OpenCode — undo/redo без файловой изоляции, у паттернов Fable 5 worktrees — рекомендация, а не механизм. Flex делает изоляцию + time-travel (/undo, /redo) встроенным свойством движка, локально и бесплатно.

**5. Rust.** Все конкуренты — TypeScript/Python. Для инфраструктурного слоя (а Flex позиционируется как инфраструктура, не продукт) это осмысленно: один бинарь, предсказуемая производительность, отсутствие рантайм-зависимостей npm/Bun — прямой контраст с supply-chain-рисками экосистем OpenClaw/OpenCode.

### Честные пробелы

*(обновлено 2026-07-09 — статусы отражают реализацию из раздела «Что позаимствовать» ниже)*

- **Зрелость и дистрибуция.** Pre-alpha против 184k–380k звёзд у конкурентов. Нет комьюнити, нет маркетплейса skills, нет истории релизов. Это самый большой разрыв, и он не закрывается архитектурой — и не закрылся: чисто продуктовый/маркетинговый пробел, вне зоны того, что можно решить кодом.
- ~~Нет durable checkpointing.~~ **Закрыто.** `SessionActor` (единственный writer на mailbox) + чекпоинты как индексированные указатели на `seq` в append-only логе. Resume по `session_id` есть; HITL-interrupt-как-состояние в духе LangGraph — нет (Flex решает эквивалентную задачу иначе, через синхронный `PermissionRequested`/`respond_permission` внутри самого турна, а не через checkpoint-based resume).
- ~~Нет client/server-разделения.~~ **Закрыто.** `flex serve` — headless HTTP/SSE с OpenAPI, bearer-auth, non-loopback-guard. Генерируемого клиентского SDK, как у OpenCode, всё ещё нет — только сама спека.
- **Gateway — задел, а не продукт.** Channel trait и каналы Telegram/Slack/Discord заявлены, но против 20+ каналов OpenClaw и Hermes это прототип. Не изменилось — P2.1 остаётся неначатым.
- **Self-learning plugin частично обкатан.** ~~Нет и независимого верификатора.~~ Независимый верификатор (`Verify`/`SubmitVerdict`, maker≠grader) и HITL-подтверждение (`require_human_approval`) есть; риск компаундирования устаревших фактов теперь имеет два независимых заслона. Не хватает главного — эксплуатации в бою (реальных сессий, статистики срабатывания гейтов).
- **Нет observability/evals-стека.** LangSmith — то, за что платят пользователи LangChain; у Devin — Session Insights и analytics. У Flex есть eval в packages/sdk, но нет tracing-продукта. Не изменилось — P2.2 остаётся неначатым.
- **Нет hosted-исполнения.** Devin (VM), Anthropic Routines, LangGraph Platform показывают спрос на laptop-off исполнение. **Частично закрыто**: Routines (`flex serve --enable-routines`, cron/webhook) дают laptop-off *исполнение*, но не laptop-off *хостинг* — раутин всё ещё нужен постоянно работающий `flex serve`-процесс где-то, serverless-исполнитель остаётся заглушкой.

## Что позаимствовать (приоритизированные рекомендации)

Приоритет 1 — критично для дифференциации; привязка к пакетам Flex указана для каждой рекомендации.

> **Статус на 2026-07-09**: все семь пунктов ниже (P1.1–P1.4, P2.3, P3.1, P3.2) реализованы.
> Только P2.1 (channel-интеграции) и P2.2 (eval-harness) остаются нетронутыми — см. отметки
> под каждым пунктом. Реализация местами разошлась с первоначальным эскизом; расхождения
> отмечены явно, без ретуши.

### P1.1. Независимый верификатор «maker ≠ grader» → packages/engine (roles + loop)
Самый подкреплённый данными паттерн исследования: fresh-context верификаторы ловят ~73% подсаженных дефектов против 7–33% у самокритики (Parameter Golf; единичный эксперимент, но направление подтверждают и дочерние Devin'ы, самостоятельно гоняющие тесты, и sandboxed-валидация Devin Security Swarm). У Flex уже есть встроенная роль reviewer в RoleSpec — усилить её контрактом: верификатор получает **только артефакты + рубрику**, никогда — рассуждения исполнителя; свежий контекст; результат верификации пишется в event stream как первоклассное событие. Это же закрывает риск компаундирования мусора в learned memory (P1.3).

**✅ Реализовано** (`996a38c`, дальнейший рефакторинг в `c07091b`): опциональный плагин `agentloop-verifier` — `Verify`-тул (loop-intercepted, спавнит `verifier`-роль с `max_depth: 0`) + `SubmitVerdict`. Вердикт (`VerificationVerdict`) идёт через уже существующий `ToolOutput.structured` — без нового wire-события. `LearningPlugin::require_verified_memory` гейтит `SkillSave`/`MemoryWrite` через `VerifiedMemoryGateHook`.

### P1.2. Checkpointer-абстракция из LangGraph → packages/engine/session (roadmap «session actor»)
Перенять именно **абстракцию**: трейт Checkpointer с реализациями in-memory/SQLite/Postgres, персистенция состояния сессии по session_id, resume повторным вызовом, HITL-interrupt как состояние сессии. JSONL ground truth Flex — уже готовый журнал для реплея; чекпоинты — это индексированные снапшоты поверх него. Учесть критику Diagrid: checkpoints ≠ durable execution — для настоящей надёжности заложить в контракт session actor'а детекцию сбоев и координацию resume (то, чего в OSS LangGraph нет и что уводит клиентов в платный Platform — для Flex это шанс).

**✅ Реализовано** (`c66481b`): `SessionActorHandle` — единственный writer на mailbox (mpsc), делающий `Subscribe` атомарным шагом относительно `Append` (закрывает документированный в AGENTS.md append→broadcast race). Чекпоинт — не отдельное хранилище, а именованный указатель на `seq` в уже существующем append-only логе (`CheckpointRef`/`CheckpointLabel`, default-provided методы на `SessionStore`). Turn-panic supervision через `tokio::spawn`+`JoinError::is_panic`. SQLite/Postgres-реализации не делались — не потребовались при JSONL-как-ground-truth дизайне.

### P1.3. Goal-loops и stop-правила из Fable-паттернов → packages/engine/loop + packages/engine/learning
Шестичастный цикл (trigger → rules load → executor → verifier → memory write → stop check) ложится на существующие hooks и learning-плагин Flex почти один в один. Конкретно: (а) условие остановки как первоклассный параметр loop (аналог /goal: done/escalate/park/go-again, потолок «3 одинаковых провала», бюджет токенов); (б) пятиступенчатая прогрессия памяти (провал → причина → проверенный факт → общее правило → консультация) как схема для MemoryWrite; (в) запись в learned memory **только через верификатор** из P1.1 — этого нет ни у Hermes, ни у OpenClaw, и это ответ на их главную слабость.

**✅ Реализовано** (`034476e`): `EngineService::run_goal(session, GoalSpec)` — `max_iterations`, `max_identical_failures` (по категории `TurnStopReason`), `token_budget`, `Achieved` (weak signal без tool-calls, либо `Verify: Pass` когда `require_verification`). **Расхождение**: `escalate`/`park` — не реализованы, оставлены зарезервированными вариантами `GoalStopReason`; наивный пост-хок скан `QuestionRequested` не отличает «нужна эскалация» от «спросили и ответили в штатном режиме», поскольку `AskUserQuestion` уже блокируется синхронно внутри самого турна.

### P1.4. Client/server split из OpenCode → packages/sdk + новый сетевой слой
Headless-сервер с OpenAPI-спецификацией и генерируемым SDK — то, что сделало OpenCode платформой, а не CLI. Для Flex: `flex serve` поверх существующего engine — event-стрим уже нормализован — осталось выставить его по HTTP/SSE. Уроки безопасности обязательны с первого дня: bind на 127.0.0.1, аутентификация по умолчанию (CVE-2026-22812 у OpenCode, ~1 000 открытых инстансов OpenClaw в Shodan, неаутентифицированный API Hermes — три одинаковых провала подряд у трёх проектов).

**✅ Реализовано** (`c07091b`): `agentloop-transport-http` — bind `127.0.0.1` по умолчанию, жёсткий отказ на non-loopback без явного токена, bearer-auth (`subtle::ConstantTimeEq`) на каждом роуте кроме `/health`, `/openapi.json` через `utoipa`. **Расхождение**: генерируемого клиентского SDK нет — только сама OpenAPI-спека; клиенты пока пишутся вручную поверх неё.

### P2.1. Channel-интеграции OpenClaw → packages/gateway
Спрос доказан (~380k звёзд OpenClaw, 20+ каналов; сервисы Tencent/Z.ai). У Flex уже есть Channel trait и задел Telegram/Slack/Discord — довести до конца именно эти три (покрывают dev-команды), скопировав из OpenClaw модель маршрутизации: входящий канал/аккаунт/пир → изолированный агент со своим workspace (у Flex — своим worktree). Дифференциация от OpenClaw — безопасность: каждый канал-агент за executor-бэкендом с network policy, injection-scan hooks на входящих сообщениях (у Flex уже есть — у OpenClaw именно этого не хватило против prompt-injection-эксфильтрации).

**⏳ Не начато.** `Channel` trait и нормализованные `Inbound`/`Outbound` остаются contract-only; Telegram/Slack/Discord-адаптеры и `ChatKey → SessionId` роутинг — следующий естественный пункт.

### P2.2. Eval-harness по мотивам Hermes/Atropos и LangSmith → packages/sdk/eval
Два образца: (а) Atropos — окружения как микросервисы, эпизодические награды за multi-turn tool-calling/code-exec (сам Atropos заархивирован 04.07.2026 — идеи свободны, конкуренция мертва); (б) LangSmith — production-трейсы становятся eval-датасетами. Для Flex естественный путь — (б): JSONL-сессии уже являются идеальными eval-фикстурами; добавить в packages/sdk/eval конвертацию «сессия → регрессионный тест» и рубрики для verifier-грейдинга (Haiku-класс моделей как дешёвые грейдеры, по образцу маршрутизации из Fable-паттернов).

**⏳ Не начато.** `packages/sdk/crates/eval` уже существует (task discovery, `MockProvider`-раннер, markdown/JSON-отчёты, baseline-гейтинг) — конвертации «сессия → регрессионная фикстура» и verifier-грейдинга в нём пока нет.

### P2.3. Организационная память Devin (Knowledge с одобрением) → packages/engine/learning
Точечный паттерн: агент **предлагает** запись в память после наблюдения повторяющегося паттерна, человек одобряет («I noticed you use React and Redux — add to Knowledge?»). Это HITL-мостик между полностью автономным self-learning Flex и enterprise-требованиями к контролю; аналог — Skill Workshop OpenClaw. Реализация: режим `learned_memory_approval: ask` в существующем precedence project>user>learned.

**✅ Реализовано, иначе, чем в эскизе** (`685b8bf`): вместо режима `ask` в precedence-конфиге — `Plugin::force_ask_tools()` (новый extension point на `core::Plugin`) + `PermissionPolicy.force_ask`, переопределяющий и `BypassPermissions`, и `DontAsk` для именованных тулов. `LearningPlugin::require_human_approval(bool)` — независимый от `require_verified_memory` булев флаг (не enum-режим, как планировалось изначально: обнаружилось, что verify-гейт и ask-гейт должны компоноваться, а не быть взаимоисключающими состояниями). Причина отхода от исходного дизайна: обычный permission-prompt на `SkillSave`/`MemoryWrite` уже существовал (`PermissionHint::Always`), но `DontAsk` тихо его отклонял, а `BypassPermissions` — тихо пропускал; именно в автономных прогонах (Routines, P3.2) человеческий чек-пойнт нужнее всего и должен быть независим от общего permission-режима сессии.

### P3.1. Dynamic Workflows → packages/engine (после стабилизации ядра)
Модель пишет собственный orchestration-harness (agent/parallel/pipeline) вместо фиксированного RoleSpec. Для Flex это дальний прицел: RoleSpec покрывает 90% случаев дешевле и предсказуемее; но выставить программируемый API оркестрации (те же примитивы, что использует RoleSpec) через SDK — низкая цена, высокая опциональность.

**✅ Реализовано, иначе, чем в эскизе** (`d6b4e5b`): `RunWorkflow` — декларативный список шагов (`task`/barrier `parallel`), не исполняемый JS-harness. Flex сознательно не даёт модели sandbox для произвольного кода — это противоречило бы принципу «loop free of process code» и было бы избыточной поверхностью атаки без явного выигрыша над существующим RoleSpec-графом. Каждый шаг выполняется через уже существующий `run_subagent()`; результат каждого шага автоматически подставляется в бриф следующего. Опционален (`EngineConfig.enable_workflow_tool`, off по умолчанию).

### P3.2. Routines / scheduled triggers → packages/engine/loop + packages/gateway
Cron/webhook/CI-триггеры как источник событий для loop (у Hermes — cron-задачи в свежих сессиях, у Anthropic — Routines, у Devin — automations с event-триггерами). Ложится на event model Flex естественно; hosted-вариант отложить до появления реального serverless-executor'а.

**✅ Реализовано** (`772c911`): `RoutineSpec`/`RoutineTrigger`/`RoutineStore` — contract-only в `packages/gateway/crates/channel` (без зависимости на `EngineService`, как и `Channel`); `RoutineRunner`/`FileRoutineStore` — в `packages/sdk` (та же причина, что у `LoopAgent`/`ClawBot`: нужен `EngineService`, которого у `gateway` умышленно нет). `flex serve --enable-routines` — 30-секундный пул для cron-триггеров (`croner` — единственная новая внешняя зависимость всего этого цикла работ) + `POST /routines/{id}/trigger` за тем же bearer-токеном. Hosted/serverless-вариант, как и предполагалось, отложен.

### Анти-рекомендации (что НЕ заимствовать)
- **Автономную публикацию/установку skills без ревью** (ClawHub): 36,8% skills с проблемами безопасности и 76 вредоносных payload'ов — прямое следствие отсутствия гейта. Precedence + подписи + HITL-одобрение обязательны для любого будущего skills-обмена.
- **Скорость релизов ценой стабильности** (OpenCode: 832 релиза, «bumpy updates», 3,6k issue) — для инфраструктурного проекта репутация стабильности важнее темпа.
- **Абстракции ради абстракций** (LangChain «abstraction bloat») — держать слои Flex (contracts/core/loop/engine) тонкими и явными.
